## Overview

An agent provider defines how Sandcastle talks to one coding agent. In the host-first Task Coordination model, it is only one part of a run:

- **Task Coordination** selects a ready **Task**
- A **backlog adapter** exposes that Task from a system such as GitHub Issues
- An **execution mode** decides where the run happens — host execution by default, sandboxed execution optionally
- An **agent provider** builds commands and parses output for the coding agent that does the work

This separation is intentional. A GitHub **Issue** is the first backlog-specific representation of a **Task**; it is not an agent-provider concern.

## Default happy path

The default happy path is:

- **Agent provider:** Pi
- **Execution mode:** host execution
- **Backlog adapter:** GitHub Issues

That makes Pi the default coding agent without collapsing Sandcastle into a Pi-only tool.

On the default host-first Task Coordination worker path backed by the GitHub Issues backlog adapter, the host also needs the `gh` CLI available because the adapter reads and updates Issue-backed Tasks through GitHub CLI. The scaffolded `.env.example` includes `GH_TOKEN` for that backlog adapter; it is not a Pi-specific requirement.

## Agent provider vs backlog adapter env

When you scaffold a worker, `.sandcastle/.env.example` is assembled from both sides of the run contract:

- the chosen **agent provider**
- the chosen **backlog adapter**

For example, the default host-first Task Coordination worker backed by the GitHub Issues backlog adapter typically includes:

- `ANTHROPIC_API_KEY` from the Pi agent provider
- `GH_TOKEN` from the GitHub Issues backlog adapter

So seeing `GH_TOKEN` in the default `github-issues-coordinator` template does not mean GitHub is part of the agent provider. It means the selected backlog adapter needs GitHub access to project Task Coordination state onto GitHub Issues.

## Custom providers

Agent providers are swappable. To add a different coding agent, implement a provider that can build commands, parse output, and supply any provider-specific env setup.

If you scaffold sandboxed execution, Sandcastle can also generate a provider-specific Dockerfile template. That Dockerfile scaffolding is only for sandboxed execution and stays separate from backlog-adapter concerns.
